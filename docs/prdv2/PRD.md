## Problem Statement

I'm landing in an unfamiliar city for a fixed number of days and I have a loose set of
interests — "ramen, museums, live jazz" — but turning that into an actual day-by-day plan is
painful. If I do it by hand I spend hours cross-referencing maps and opening hours; if I ask a
generic chatbot it suggests places that are closed or don't exist and routes me back and forth
across the city with no regard for travel time. I want real places I'd actually like, and a
plan that respects where things are so I'm not wasting half the trip in transit.

## Solution

A locally-run planner that I drive in stages. I set up a Trip (city, when I land, how many
days, my pace, my lodging) and pick Categories with the specific Interests I care about. The
planner proposes concrete, real **Options** for each Interest, which I research and rate 1–5
stars. Once I've rated enough, it generates a **Schedule**: my Chosen Options laid out day by
day, each day starting and ending at my Home Base, ordered to minimise travel and clustered so
that being in one area pulls in nearby things. It tells me the travel time between stops, warns
me if I've picked more than the trip can hold, and never silently invents or drops places. It
runs on free LLM tiers and keeps working when one runs out.

## User Stories

**Trip setup**
1. As a traveller, I want to create a Trip with a city, arrival time, and number of days, so that the plan is bounded to my actual trip.
2. As a traveller, I want to set my pace (relaxed / balanced / packed), so that days aren't over- or under-filled for my taste.
3. As a traveller, I want to set a daily start/end window, so that nothing is scheduled when I want to be resting.
4. As a traveller, I want to choose a transport mode (walk / transit / drive), so that travel times reflect how I'll actually get around.
5. As a traveller, I want to set my lodging as a Home Base, so that each day starts and ends where I'm staying and that travel is counted.
6. As a traveller, I want the planner to offer common default Categories, so that I don't start from a blank page.
7. As a traveller, I want to add my own Categories and type specific Interests under each, so that the planner searches for exactly what I care about.
8. As a traveller, I want my Trip saved locally and resumable, so that I can plan across several sittings without losing work.

**Gathering Options**
9. As a traveller, I want concrete, named Options proposed for each Interest, so that I have specific places to research rather than vague ideas.
10. As a traveller, I want Options to be real and current (grounded), so that I don't waste time on places that have closed or never existed.
11. As a traveller, I want each Option to come with a description, address, and rough visit duration, so that I can judge it quickly.
12. As a traveller, I want re-running a gather for the same city/Interest to be instant and free, so that experimenting doesn't burn tokens.
13. As a traveller, I want duplicate Options across Interests collapsed, so that the same place doesn't appear twice.
14. As a traveller, I want to add a place I already have in mind, so that my own must-visits are part of the plan.

**Rating & capacity**
15. As a traveller, I want to rate each Option 1–5 stars, so that the planner prioritises what I like.
16. As a traveller, I want anything I rate below 3 excluded, so that weak Options never show up in my Schedule.
17. As a traveller, I want my top-rated Options favoured when not everything fits, so that the trade-offs reflect my preferences.
18. As a traveller, I want to be told when I've chosen more places than the trip can hold, so that I can add a day, raise the pace, or accept some being dropped.
19. As a traveller, I want to be told when I have too few, so that I can gather more before generating a thin Schedule.
20. As a traveller, I want the "enough options" check to be advice, not a gate, so that I can always proceed when I want to.

**Scheduling**
21. As a traveller, I want my Chosen Options laid out across the days, so that I have a concrete itinerary.
22. As a traveller, I want each day ordered to minimise travel, so that I'm not zig-zagging across the city.
23. As a traveller, I want being in an area to pull in nearby Options, so that a long stop (e.g. a theme park) is surrounded by things in that area.
24. As a traveller, I want food placed at meal times and everything within opening hours, so that the plan is actually doable.
25. As a traveller, I want Day 1 to begin when I land, so that a late arrival doesn't pretend I have a full first day.
26. As a traveller, I want to see the travel time between consecutive stops, so that I can trust and follow the route.
27. As a traveller, I want a fixed commitment (a wedding, a timed ticket, a flight) placed at its exact time with the day built around it, so that I never miss it.
28. As a traveller, I want regenerating the Schedule to leave my fixed commitments untouched, so that locking something means it stays put.
29. As a traveller, I want to be warned clearly when a must-do can't fit (e.g. it's only open on a day I'm not there), so that I'm not surprised by a silent omission.
30. As a traveller, I want a readable day-by-day view, optionally with a friendly summary, so that I can actually use the plan.

**Reliability, cost & diagnostics**
31. As a traveller on a free LLM tier, I want the planner to switch providers when one is exhausted, so that planning doesn't stop mid-trip.
32. As a traveller, I want to run entirely on a local model if I have no API key, so that I can use the tool without signing up for anything.
33. As a developer, I want every run to carry a request id I can see, so that I can trace what happened.
34. As a developer, I want logs and token usage queryable by that id, so that I can debug behaviour and watch cost.
35. As a traveller, I want to manage several Trips (e.g. June and July), so that planning a new trip doesn't disturb another.

## Implementation Decisions

- **UI-agnostic core, CLI first.** All logic lives in a core package consumed by a thin CLI adapter now and a FastAPI+React adapter later; the core never depends on an adapter ([ADR-0002](../adr/0002-ui-agnostic-core-cli-first.md)).
- **The LLM is abstracted behind a provider interface with a fallback chain.** Gemini → Groq/OpenRouter → local Ollama; a quota/rate-limit error falls through to the next provider. Grounding is decoupled from the model (a Gemini capability, optionally a separate search step) so non-grounding models still work ([ADR-0006](../adr/0006-llm-provider-abstraction-fallback-chain.md)).
- **Geography is resolved through a tiered provider, never trusted to the LLM.** Geocoding cascades Google Places → OpenStreetMap (Nominatim) → the LLM's own coordinate estimate; travel times cascade Google Distance Matrix → OSRM → haversine. Each Option records which tier answered ([ADR-0001](../adr/0001-tiered-geo-sourcing.md)).
- **Scheduling is done by the LLM but constrained and checked.** The model receives the Chosen Options, the real travel-time matrix, the Home Base, any Pinned Commitments, and the constraints/preference weights, and returns a structured Schedule. A deterministic validator scores it (opening-hours/window/meal violations, per-day travel/zig-zag, pins-unmoved, days-anchored-at-Home-Base) and triggers a capped re-prompt when it scores poorly. The validator — not the model — is the quality contract ([ADR-0005](../adr/0005-deterministic-travel-aware-scheduler.md)).
- **Anchors.** Home Base anchors the start/end of every day; Pinned Commitments are fixed time+place items (one concept covering weddings, locked tickets, flights) that the scheduler routes around and never moves on regeneration; the scheduler is anchor-aware (clusters nearby Options around fixed points) ([ADR-0007](../adr/0007-anchors-and-anchor-aware-scheduling.md)).
- **Ranking & capacity.** Rating is 1–5 stars; Chosen = rated ≥ 3; the highest ratings are force-included on oversupply. Capacity = days × slots-per-day (by pace) minus a partial Day 1; the gate is advisory, never blocking.
- **Token economy.** Option-gathering is batched one call per Category, cached by (city, Interest), and only Chosen Options reach geocoding and scheduling. Grounded gather requests JSON in-prompt with defensive parsing + a repair retry (grounding and strict response-schema may not compose in one call).
- **Two stores.** Domain state (Trip, Category, Interest, Option, ratings, Schedule) in SQLite behind a repository interface; logs/telemetry in Postgres ([ADR-0004](../adr/0004-sqlite-state-postgres-logs.md)).
- **Observability.** OpenTelemetry SDK with a custom Postgres exporter; the OTel `trace_id` is the request id, returned later as an `X-Request-ID` header by the FastAPI layer ([ADR-0003](../adr/0003-otel-with-postgres-exporter.md)).

Decision-encoding shapes (from the design, not implementation code):

```
Option:        name, description, address, source(llm|user), lat, lng,
               geo_source(google|osm|llm_estimate), opening_hours,
               est_visit_minutes, user_visit_minutes?, image_ref?,
               rating(1..5)?  →  chosen = rating ≥ 3
               visit_minutes  = user_visit_minutes or est_visit_minutes
PinnedCommitment:  title, lat, lng, start, end?, kind(event|ticket|flight)   # immovable
Trip:          city, arrival_at, num_days, pace, daily_start, daily_end,
               transport_mode, home_base, pins[]
Telemetry:     log_event(ts, level, request_id, trip_id, component, payload_jsonb, latency_ms)
               llm_call(ts, request_id, trip_id, provider, model,
                        prompt_tokens, completion_tokens, latency_ms, cache_hit)
```

## Testing Decisions

- **Test external behaviour, not implementation.** Because the LLM scheduler is
  non-deterministic, tests assert *properties* of outputs (no opening-hours violation, capacity
  respected, pins unmoved, days anchored at Home Base) rather than exact layouts.
- **Seams (highest available; confirm these match expectations).** Test through the core's
  use-case functions (the gather / rate / capacity / schedule steps), faking the external
  effects at the **provider interfaces** — the LLM provider and the geo/travel providers — and
  using an in-memory database at the **repository interface**. The **validator** is a pure
  function tested directly. These are existing seams in the design; no new seams are needed.
- **Default runs hit no network.** Hand-written fakes for the LLM and geo providers make the
  whole pipeline deterministic and free to run.
- **Property-based tests** (Hypothesis) generate random Options/constraints and assert the
  validator's invariants always hold.
- **A few recorded-payload tests** catch provider schema drift; **live tests** against real
  Gemini/Maps are gated behind an env flag and never run by default.
- **Prior art:** none yet — this is a greenfield repo, so these tests establish the patterns.

## Out of Scope

- Booking, payments, or reservations.
- Full flight/transfer modelling (airport transfers, mid-trip flights) and multi-city trips —
  the Schedule covers a single city and starts when the traveller lands.
- Accounts, multi-user, collaboration, or sharing.
- The non-technical-user experience (setup wizard, packaged app, JSON storage, per-OS docs) —
  documented as a goal, deferred.
- Web UI concerns: images, an interactive map, drag/delete editing — these come with the
  FastAPI+React phase.
- In-trip live re-routing; weather/budget-aware planning (candidate future features).

  tiers; the fallback chain and the (city, Interest) cache exist for this.
